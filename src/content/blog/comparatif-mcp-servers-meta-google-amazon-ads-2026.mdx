<SectionHeader number="01" title="Pourquoi un comparatif des MCP servers ad tech maintenant" />

Entre octobre 2025 et avril 2026, **les trois plus grandes plateformes pub mondiales ont sorti leurs serveurs MCP officiels**. Cumulé à Google Analytics (juillet 2025), ça donne un écosystème ad tech "agentique" qui devient opérationnel pour les opérateurs growth en 2026.

Le standard MCP (Model Context Protocol) a été créé par Anthropic en novembre 2024 puis transféré à la Linux Foundation. Il fonctionne comme une couche de communication universelle entre un agent IA et un système externe. L'intérêt : un seul agent (Claude, ChatGPT, Gemini) peut interroger plusieurs plateformes simultanément si chacune publie son serveur MCP.

C'est précisément ce qui se passe. **Le paysage est devenu cohérent**, et ça change la manière dont un Traffic Manager DNVB pilote son compte au quotidien.

<Callout variant="insight" title="Le sujet n'est plus émergent">
La recherche francophone autour des MCP ad tech a explosé depuis l'automne 2025. "Google Ads MCP", "agent IA marketing", "Claude Code Meta" sont passés en quelques mois de termes confidentiels à requêtes actives des opérateurs growth qui veulent comprendre comment intégrer ces outils dans leur stack.
</Callout>

Ce comparatif fait le tri sur trois questions concrètes :
1. Qu'est-ce que chaque MCP fait vraiment (et ne fait pas) ?
2. Lequel brancher en premier selon le profil DNVB ?
3. Quelle est la valeur résiduelle des outils SaaS positionnés "AI pour Meta Ads" maintenant que MCP existe ?

<SectionHeader number="03" title="Google Ads MCP : le pionnier read-only" />

Google a été la première grande plateforme pub à lancer un serveur MCP officiel, **le 7 octobre 2025**. C'est cohérent avec leur stratégie de mise à disposition open source de leurs outils API.

**Ce qu'il fait concrètement.**

Le serveur expose **3 tools** :
- `list_accessible_customers` : retourne la liste des comptes Google Ads accessibles à l'utilisateur authentifié
- `search` : exécute des requêtes en GAQL (Google Ads Query Language), le langage natif de l'API Google Ads, pour récupérer metrics, budgets, statuts
- `get_resource_metadata` : retourne la structure des objets Google Ads (campaign, ad_group, ad, etc.) pour savoir quoi requêter

**Ce qui le différencie.**

Mode **read-only** à ce jour (documentation officielle Google Developers, mise à jour 28 avril 2026). Tu peux interroger ton compte, pas le modifier via le MCP officiel.

**Open source** : le code est sur `github.com/googleads/google-ads-mcp`. Tu peux l'auto-héberger, le forker, l'auditer. C'est le seul des trois MCP majeurs qui offre ce niveau de transparence.

**Bas niveau** : les 3 tools sont volontairement minimalistes. Le développeur ou l'agent IA construit sa propre logique métier au-dessus. C'est puissant si tu sais ce que tu fais, plus brut si tu cherches du plug-and-play.

<Callout variant="insight" title="Le piège du developer token Google Ads">
Avant de connecter un MCP Google Ads, il te faut un **developer token** (chaîne de 22 caractères) délivré par Google après validation de ton app dans la Google Ads API console. Pas instantané. Si tu démarres, prévoir 5 à 10 jours ouvrés entre la demande et l'approbation. C'est ce qui rend l'expérience plus friction que Meta MCP (qui ne demande pas de token développeur).
</Callout>

**Setup minimal (configuration JSON dans ton client MCP) :**

```json
{
  "mcpServers": {
    "google-ads-mcp": {
      "command": "pipx",
      "args": ["run", "--spec", "git+https://github.com/googleads/google-ads-mcp.git", "google-ads-mcp"],
      "env": {
        "GOOGLE_PROJECT_ID": "YOUR_PROJECT_ID",
        "GOOGLE_ADS_DEVELOPER_TOKEN": "YOUR_DEVELOPER_TOKEN"
      }
    }
  }
}
```

**Pour qui c'est utile en priorité.**

Équipes data et eng qui font du reporting SEA en SQL, agences qui auditent plusieurs comptes Google Ads. Pas le meilleur choix pour un Traffic Manager DNVB qui veut juste piloter son compte en conversation simple.

<SectionHeader number="04" title="Amazon Ads MCP : le pragmatique avec Skills pré-cuits" />

Amazon a annoncé l'open beta de son MCP server le **2 février 2026** (après une closed beta démarrée le 13 novembre 2025). L'approche est différente de Google : plus de packaging, moins de bricolage.

**Ce qu'il fait concrètement.**

L'architecture Amazon expose **deux types de primitives** :
- **Tools** : actions atomiques (créer/modifier/supprimer une campagne, lire des metrics, générer un rapport, accéder à la facturation)
- **Skills** : workflows pré-cuits définis par Amazon Ads qui orchestrent plusieurs tools pour exécuter une tâche complète

**L'innovation : les Skills.**

Concrètement, un Skill type "expand campaign to new country" prend un prompt comme `"Add UK to [campaign_id] with a £10 budget"` et exécute en arrière-plan plusieurs appels API coordonnés pour faire le travail (création du marché, allocation budget, settings localisés). L'agent IA n'a pas à orchestrer la séquence, c'est Amazon qui fournit la recette.

<PullQuote attribution="Paula Despins, VP Ads Measurement chez Amazon Ads (annonce officielle, 2 février 2026)">
Tools reduce complexity by orchestrating Amazon Ads capabilities into complete, multi-step operations that are core to everyday advertising workflows.
</PullQuote>

**Les Skills documentés au lancement** :
- Création d'un compte
- Génération de rapports
- Lancement de campagne end-to-end (Sponsored Products en un prompt)
- Expansion d'une campagne vers un nouveau marché géographique

**Clients IA compatibles** : Claude, ChatGPT, Gemini, plus l'écosystème AWS (Kiro, Bedrock, AgentCore).

**Pour qui c'est utile en priorité.**

DNVB qui font du Amazon comme canal significatif (>20% du CA). Marques internationales qui doivent réplicer des structures de campagne sur plusieurs marketplaces. Beaucoup moins pertinent pour une DNVB Meta-first qui ne fait pas ou peu d'Amazon.

<SectionHeader number="05" title="Meta Ads MCP : le plus complet pour les DNVB Meta-first" />

Meta a lancé en dernier, **le 29 avril 2026**, mais avec le scope fonctionnel le plus large à ce jour.

**Ce qu'il fait concrètement.**

Quatre **capabilities** macro selon l'annonce officielle :
- **Comprehensive Reporting** : performances de campagne, extraction de rapports détaillés
- **Campaign Management** : création et édition de campaigns, ad sets, ads en langage naturel
- **Catalog Management** : création de catalogue produit, ajout de produits, debug des feeds
- **Signal Diagnostics** : lecture de la santé du Pixel et de la Conversions API

**Plus un CLI complémentaire** pour les usages scriptés (jobs récurrents, pipelines de reporting automatisés, intégrations internes).

**Ce qui le distingue.**

Le packaging Meta est intermédiaire entre Google et Amazon. Pas de Skills pré-cuits comme Amazon, mais des capabilities macro plus larges que les 3 tools bas niveau de Google. C'est un compromis pragmatique qui correspond au profil de ses utilisateurs cibles : opérateurs growth qui veulent un outil utile sans demander un investissement dev.

**Setup encore plus minimaliste que Google :**

Pas de developer token, pas de configuration JSON manuelle, pas de validation d'app Meta Developers. Tu autorises depuis le client MCP, Meta gère l'authentification.

**Pour qui c'est utile en priorité.**

Toute DNVB où Meta représente une part significative du spend. C'est le MCP avec la friction de setup la plus basse et le scope fonctionnel le plus large. Si tu choisis un seul MCP ad tech à brancher, c'est lui pour 80% des opérateurs DNVB.

<SectionHeader number="07" title="L'avenir : Google Analytics, TikTok, LinkedIn et l'ad tech 'agentique'" />

Au-delà des trois plateformes ads majeures, l'écosystème MCP ad tech s'étend rapidement.

**Déjà actifs (mai 2026) :**
- **Google Analytics MCP** : lancé le 22 juillet 2025, ouvre la requête GA4 en langage naturel. Critique pour la mesure cross-canal.
- **Microsoft Clarity MCP** : lancé le 4 juin 2025, web analytics conversationnel.
- **AppsFlyer MCP** : lancé le 17 juillet 2025, mesure mobile cross-canal.
- **Ad Context Protocol** : initiative IAB avec Scope3, Yahoo, PubMatic, Swivel, Triton Digital, Optable (octobre 2025).

**Attendus / en discussion :**
- **TikTok Ads MCP** : pas d'annonce officielle à ce jour. La structure technique TikTok Marketing API se prête bien à un MCP. Probable arrivée fin 2026 ou 2027.
- **LinkedIn Ads MCP** : pas d'annonce officielle. LinkedIn fait partie de l'écosystème Microsoft (qui a Clarity MCP), une intégration est probable mais non confirmée.
- **Pinterest / Snapchat Ads MCP** : volumes pub plus faibles, priorité plateforme probablement plus basse.

**Le pattern macro qui se confirme.**

L'ad tech est en train de standardiser son interface IA sur MCP. Dans 12 à 18 mois, **avoir un MCP officiel sera la norme** pour toute plateforme pub crédible. Les outils SaaS qui se positionnent comme "connecteurs vers Meta/Google" perdent leur valeur basse. Les outils qui survivent et croissent se positionnent plus haut dans la stack :
- Modèles de décision propriétaires (bid IA, allocation budget cross-canal optimisée)
- Production créative IA personnalisée par marque et verticale
- Données enrichies non disponibles via les API natives
- Expertise verticale (DNVB, B2B, marketplace, fintech)

C'est cohérent avec l'évolution naturelle du marché : la commodité descend, la valeur monte. Le MCP est la commodité de demain. Ce qu'on construit au-dessus, c'est la valeur.

Pour une DNVB qui veut industrialiser le pilotage Meta sans construire elle-même son agent IA et sa stack créative, l'angle Gang4 reste valide : on industrialise la production créative UGC + diffusion via whitelisting Partnership Ads, avec un agent IA qui peut piloter Meta via le MCP officiel et utiliser nos créas comme input. Le MCP automatise la couche pilotage. Le pipeline créatif reste l'enjeu principal de performance, et c'est là que [le framework PDA](/blog/framework-pda-creas-meta-2026/), [le funnel TOFU/MOFU/BOFU](/blog/funnel-meta-ads-tofu-mofu-bofu-dnvb/) et [Andromeda](/blog/andromeda-meta-algorithme-delivery/) restent les vraies leviers de scaling.
