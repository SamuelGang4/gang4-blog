<SectionHeader number="02" title="Le risque réel : ban de compte + learning phase cassée" eyebrow="Avant de toucher quoi que ce soit" />

Avant d'entrer dans les system prompts et les workflows, il faut comprendre les deux risques concrets que portent un agent IA mal encadré sur Meta Ads.

**Risque 1 : le ban de compte.**

Meta applique ses Marketing API Terms strictement. Les automatisations qui contournent l'interface officielle ou qui sollicitent l'API à des rythmes anormaux peuvent déclencher des protections anti-abus. Plusieurs cas documentés sur Reddit r/FacebookAds montrent des comptes bannis **définitivement** suite à des usages aggressifs de Claude Code (modification de budgets toutes les 2 heures, pause/relance en boucle de créatifs, scripts qui interrogent l'API en continu).

<PullQuote attribution="Thread Reddit r/FacebookAds, repris dans les SERP FR (mars 2026)">
Si tu utilises Claude Code pour faire du day-trading avec les annonces Meta, tu vas être banni. Utilise Claude Code avec précaution.
</PullQuote>

**La règle absolue qui évite ce risque** : utiliser **le MCP officiel Meta** (lancé le 29 avril 2026) plutôt que des scripts qui automatisent l'interface Ads Manager ou qui passent par des proxys. Le MCP officiel est Meta-authentifié, autorisé par Meta, et explicitement conforme aux Marketing API Terms.

Cf. [notre guide complet sur Meta Ads MCP](/blog/meta-ads-mcp-piloter-campagnes-claude-chatgpt/) pour le setup officiel.

**Risque 2 : la learning phase cassée.**

C'est le piège plus subtil mais tout aussi coûteux. Quand tu modifies substantiellement un ad set Meta (budget, audience, bid, créatif), Andromeda replace cet ad set en learning phase. La phase dure environ 50 événements d'optimisation. Pendant cette phase, le delivery est instable, le CPA fluctue, les performances sont imprévisibles.

Un agent IA qui a accès en écriture et qui modifie trop fréquemment les ad sets (parce qu'il optimise sur un signal court terme) déclenche des **resets de learning phase en cascade**. Ton compte reste bloqué en apprentissage permanent. La performance s'effondre en 5 à 7 jours, sans signal d'alerte évident pour un Traffic Manager qui ne suivrait pas le compte au jour le jour.

[Andromeda](/blog/andromeda-meta-algorithme-delivery/) est conçu pour optimiser sur des fenêtres stables. Un agent qui le perturbe sans le savoir détruit la valeur que l'algorithme apporte.

**Le coût opérationnel cumulé.**

Un ban de compte = restart from scratch sur un nouveau Business Manager. Plusieurs semaines pour récupérer le niveau de performance précédent. Une learning phase cassée = budget gaspillé pendant 1 à 3 semaines sur des delivery aléatoires. Les deux scénarios coûtent cher. Les deux sont évitables avec un encadrement explicite.

<SectionHeader number="04" title="Le system prompt opérationnel pour piloter Meta Ads" eyebrow="Template prêt à copier" />

Le system prompt est le contrat entre toi et Claude. Il définit ce que l'agent peut faire, ne peut pas faire, et comment il doit raisonner. Sans system prompt explicite, Claude utilise ses défauts génériques qui ne sont pas adaptés au contexte Meta Ads.

**Template structuré en 4 blocs.**

```
[BLOC 1 · CONTEXTE]
Tu es un assistant d'analyse Meta Ads pour une DNVB e-commerce. Tu n'es PAS un
agent autonome de pilotage. Tu opères en mode copilote : tu analyses, tu
suggères, tu exécutes uniquement après confirmation humaine explicite.

[BLOC 2 · PERMISSIONS]
- LECTURE : autorisée sans restriction (reporting, analyse, audit de structure,
  diagnostic créatif)
- ÉCRITURE : INTERDITE par défaut. Pour toute action en écriture (modification
  de campagne, ad set, créatif, budget, audience), tu demandes confirmation
  humaine explicite avant exécution et tu attends la réponse "OK" avant d'agir.

[BLOC 3 · GARDE-FOUS]
- Tu ne modifies pas un ad set qui a été modifié il y a moins de 7 jours, sauf
  si la modification est explicitement marquée comme "urgence critique
  documentée" par l'opérateur humain.
- Tout changement de budget supérieur à 20% en absolu, ou de bid supérieur à
  15%, demande confirmation humaine.
- Toute pause ou relance d'un même créatif dans la même semaine est interdite
  sans validation.
- Tu ne fais jamais d'optimisation "autonome continue" (style day-trading sur
  les ad sets). Tu rapportes, tu suggères, tu n'agis pas sans demande humaine.

[BLOC 4 · STYLE & ANTI-HALLUCINATION]
- Tu cites TOUJOURS la metric source et la période exacte de tes analyses.
- Si tu n'as pas la donnée, tu le dis explicitement : "donnée non disponible".
- Tu ne fabriques pas de chiffres plausibles : si l'API ne renvoie pas la
  data, tu signales la limite.
- Pour chaque insight, tu fournis le "show me the raw data" sur demande.
- Tu utilises le tutoiement et un ton direct (style opérateur growth, pas
  consultant).
```

**Pourquoi chaque bloc compte.**

Le bloc 1 cadre Claude comme un copilote, pas un autopilote. C'est ce qui fait la différence entre un usage productif et un compte banni. Le bloc 2 sépare explicitement les permissions lecture (sans risque) et écriture (à valider). Le bloc 3 inscrit la règle Meta sur la learning phase et les seuils de changement substantiels. Le bloc 4 lutte contre les hallucinations qui ont fait perdre du temps à plusieurs opérateurs (cf. les retours communautaires sur Manus AI documentés par PPC Land).

<Callout variant="tip" title="Où placer le system prompt">
Dans Claude Desktop, le system prompt se met dans les Projects (un projet dédié "Meta Ads Pilotage" avec ce prompt en instructions persistantes). Dans Claude Code, place-le dans un fichier `CLAUDE.md` à la racine de ton dossier projet. Claude Code le lit automatiquement comme contexte persistant à chaque session.
</Callout>

**Le piège qui revient le plus souvent.**

Les opérateurs qui se font avoir partagent un pattern : ils activent l'écriture trop tôt, avant d'avoir testé le comportement de Claude en read-only. Première semaine, tout semble fonctionner. La règle du "2 à 4 semaines en read-only" n'est pas une précaution excessive. C'est la fenêtre nécessaire pour calibrer ton jugement sur ce que Claude fait bien et ce qu'il fait mal.

<SectionHeader number="07" title="Anthropic best practices appliquées au pilotage Meta Ads" eyebrow="Officiel" />

Anthropic publie une documentation officielle sur les best practices de prompting Claude. Voici comment elles s'appliquent au contexte Meta Ads.

**Principe 1 : être spécifique plutôt que général.**

Mauvais prompt : "Analyse mon compte Meta et donne-moi des recommandations". Bon prompt : "Analyse les 30 derniers jours sur mes 3 campagnes TOFU broad, sors-moi le CPA par créatif et identifie les 3 créatifs avec le meilleur ratio commentaires/likes". Le second prompt donne 80% de chances de produire du code et de l'analyse utiles dès la première tentative.

**Principe 2 : utiliser des exemples (few-shot).**

Pour les analyses récurrentes, donne à Claude un exemple de format de sortie souhaité avant la requête. Exemple : "Voici comment je veux mon reporting hebdo formatté : [exemple JSON]. Génère le reporting de la semaine du X au Y dans ce format". Réduit drastiquement les itérations.

**Principe 3 : structurer avec XML pour les prompts longs.**

Anthropic recommande l'usage de balises XML pour structurer les prompts complexes. Exemple :

```
<contexte>
DNVB beauté, scaling de 30k€ à 80k€/mois sur Meta, 6 campagnes actives.
</contexte>

<demande>
Audit complet de structure + flagging des inefficacités.
</demande>

<output_attendu>
Format JSON avec champs : campaign_id, issue_type, severity, recommendation.
</output_attendu>
```

Claude lit les balises XML comme des sections nettes. La précision augmente.

**Principe 4 : laisser Claude penser (extended thinking).**

Pour les analyses complexes, demande à Claude de réfléchir avant de répondre. "Avant de proposer ta recommandation, raisonne étape par étape : (1) quels sont les patterns dans la data, (2) quelles hypothèses peuvent expliquer ces patterns, (3) quelle action serait la plus testable". L'output est plus solide et tu peux suivre le raisonnement.

**Principe 5 : être explicite sur ce qui ne doit pas être fait.**

Au-delà des permissions positives ("tu peux faire X"), inscris les interdictions explicites ("tu ne fais jamais Y"). Claude respecte mieux les interdictions formulées explicitement que les inférences depuis le contexte.

<PullQuote attribution="Anthropic, Prompting best practices documentation">
The most effective prompts are clear, specific, and provide enough context for Claude to understand both what you want and what you don't want.
</PullQuote>
